## 发现的问题

### P0/P1 候选：编辑标题后本地 Markdown 可能产生旧文件残留

- 位置：
  - `db.js:568`
  - `db.js:570`
  - `db.js:572`
  - `db.js:574`
  - `panel.js:1352`
  - `db.js:769`
  - `db.js:775`
- 现象：`writeArticleToLocalFolder()` 的文件名包含标题：`日期_标题_shortId.md`。C1 编辑器允许修改 H1 标题，保存后会按新标题创建新文件，但旧标题文件不会被删除。
- 风险：本地文件夹里可能同时存在旧标题和新标题两个 `.md`，且二者 shortId 相同。后续重新扫描时，扫描顺序不稳定，旧文件可能把新标题或新分析内容覆盖回 IndexedDB。
- 建议修复：
  - 写本地文件时按 shortId 查找同一记录的旧 `.md`，删除或覆盖旧文件。
  - 或改成稳定文件名策略，文件名只依赖日期和 shortId，不依赖标题。
  - 至少在 C1 保存标题变化时，不应留下同 shortId 的旧文件。

### P1：重新扫描会把未变化文件也计为已同步

- 位置：
  - `db.js:722`
  - `db.js:727`
  - `db.js:739`
  - `db.js:772`
  - `db.js:775`
  - `db.js:777`
- 现象：`mergeParsedMarkdownRecord()` 只要解析到 H1 标题，就会设置 `articlePatch.title`，于是 `hasPatch = true`。这意味着即使标题和原记录完全相同，`syncFromLocalFolder()` 也会 `dbPutArticle()` 并让 `synced++`。
- 风险：用户每次点击「重新扫描」，可能看到所有文件都显示“已同步”，无法判断是否真的有改动。
- 建议修复：
  - `mergeParsedMarkdownRecord()` 应比较 patch 与 existing，只有值真的变化时才设置 `hasPatch`。
  - `synced` 只统计真实变更，不统计无变化重写。

### P1/P2：格式异常的 Markdown 保存会被规范化覆盖

- 位置：
  - `panel.js:1339`
  - `panel.js:1348`
  - `panel.js:1352`
  - `panel.js:1355`
- 现象：用户在编辑器里写了格式异常的 `## 知识节点` 行时，解析器会保留旧 `nodes_hit`，这是安全的。但 `saveMarkdownEdit()` 会先写数据库、再 `writeArticleToLocalFolder(merged.record)`，之后才显示 warning。
- 风险：用户刚写的异常节点行会被规范化后的旧节点内容覆盖，可能感觉“我写的东西突然没了”。数据安全没破，但交互上不够诚实。
- 建议修复：
  - 如果存在 warning，先不要写回本地文件，或要求用户确认“只保存可解析内容”。
  - 更好的是在保存前显示 warning，让用户知道哪些段落不会生效。

### P2：Markdown 编辑器仍不是完整 Markdown 编辑器

- 位置：
  - `db.js:605`
  - `db.js:677`
- 现象：普通 section 仍按 `## 标题` 切段。比如用户在 `## 编辑评语` 里再写一个二级标题，后续内容不会作为评语保存。
- 风险：C1 完成报告已经说明这是 textarea 轻量版，但真实使用时仍可能和“Markdown 编辑器”的直觉有差距。
- 建议修复：
  - 后续 C1.1 明确哪些段落可编辑。
  - 或对 `编辑评语` 这种自由文本段落采用 tail/更明确的结束规则。
