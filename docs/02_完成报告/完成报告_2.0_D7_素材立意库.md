# 完成报告 2.0 D7：素材 / 立意库

完成时间：2026-05-08  
执行依据：用户 D7 指令 + `任务下发_2.0冲刺_九日清单.md` T-6.1 / T-6.2 / T-6.3 / T-6.4  
风险级别：P0 数据层改动。已做 v3→v4 migration 测试，确认旧 `articles / nodes / config` 不丢。

## 做了什么

1. IndexedDB `DB_VERSION` 从 3 升到 4。
2. 新增 `clips` object store：
   - keyPath: `id`
   - index: `by_type`
   - index: `by_article`
   - index: `by_savedAt`
3. 新增 clips API：
   - `dbSaveClip(clip)`
   - `dbGetClips(filter)`
   - `dbDeleteClip(id)`
   - `dbDeleteClipsBySource(articleId)`
   - `dbReplaceClipsBySource(articleId, clips)`
4. `dbDeleteArticle(id)` 改为在同一事务中删除文章、清理 nodes 引用、清理该文章的 clips。
5. `dbClearAll()` 同步清空 `clips`。
6. `dbExportAll()` 改为导出 clips，`export_format` 升到 3。
7. `dbImportAll()` 兼容导入 clips，并把备份里的 `source_article_id` 映射到实际保存后的 article id。
8. 分析保存成功后，把 `analysis.reusable_clips` 转成「素材」，把 `analysis.essence_insights` 转成「立意」入库。
9. clip 入库发生在 `dbSaveArticle()` 和 `dbReplaceArticleNodes()` 成功之后；同一来源文章的 clips 使用 `dbReplaceClipsBySource()` 一次事务替换，避免重复沉淀。
10. 资产 Tab 增加三段入口：`素材 / 立意 / 作品`。
11. 资产 Tab 默认进入「素材」，搜索框改为「搜素材、立意、人物、角度…」。
12. 「素材 / 立意」卡片显示内容、复用理由和来源文章；来源文章可点击进入资产详情。
13. 测试 runner 增加异步 keepalive，并把 `Suite.run()` 改成 module top-level await 调用，保证 IndexedDB migration 测试能真实跑完，不再被 headless 过早 dump。

## Schema Migration

迁移路径：

```js
if (old < 4) {
  ensureClipsStore(db);
}
```

`ensureClipsStore(db)` 会先检查 `db.objectStoreNames.contains('clips')`，不存在才创建，避免重复创建报错。

真实测试构造了一个 legacy v3 数据库，包含：

- 1 条 article
- 1 条 node
- 1 条 config.projects
- 无 clips store

然后用当前 `dbOpen()` 升级到 v4，验收结果：

```text
articles.length === 1
nodes.length === 1
projects[0].id === 'default'
dbGetClips() 可调用且返回 []
```

## 风险预警

1. 这是 schema 改动，必须让 Claude 用真实旧库再验一次 v3→v4 升级。
2. clips 导出是 JSON 备份格式的向前扩展；旧备份仍可导入，新备份多一个 `clips` 字段。
3. `dbReplaceClipsBySource()` 会替换同一文章的旧 clips，避免重新分析后重复沉淀；这符合“按来源文章重建素材”的逻辑。
4. 真实 API 分析后是否产生高质量素材，取决于 D6 prompt 输出质量；D7 只保证入库和展示链路。
