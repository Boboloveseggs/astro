## 一、背景：1.0 与 2.0 的关系

### 用户产品愿景
- **1.0**（`C:\参赛文档\zhijing_radar_v3_final_package\`）：先做外观（深色金色主题、3D 认知地球、时间轴卡片）再补功能 → 视觉惊艳但功能逻辑混乱、节点重叠 bug

- **本轮目标**：把 1.0 的视觉资产移植到 2.0，并新增创作者向的功能闭环。**只增不删**。

### 用户产品定位（一句话）
> **服务于创作者本人，直接加入到日常创作环节中**：抓取知乎内容 → AI 创作者向分析 → 追问深化 → 归档到认知地图 → 周复盘指导下次创作。

### T-2.2 节点位置：纯 Fibonacci sphere 防重叠

**影响文件**：`panel.js`

**核心算法**（写在 `sphereVectorForSeed(name, totalNodes, slotIndex)`）：
```js
function sphereVectorForSeed(name, total, slot) {
  const golden = Math.PI * (3 - Math.sqrt(5));
  const t = (slot + 0.5) / Math.max(1, total);
  const lat = Math.asin(1 - 2 * t) * 180 / Math.PI;
  const lon = ((slot * golden) * 180 / Math.PI) % 360 - 180;
  return spherePoint(lat, lon);
}
```

**关键要求**：
1. **不要使用任何手工经纬度表**（1.0 的 `namedPoints` 和 `mainOrder` 都丢掉）—— 这是 1.0 节点重叠 bug 的根因
2. 节点列表按确定性顺序排（建议按 node.id 字典序），让节点位置稳定
3. 总节点数 N 变化时，所有位置重新计算 → 永远均匀分布
4. 节点数 ≥ 100 时仍保证不重叠（数学已保证）

**验收**：
- 录入 5 个节点：分布均匀
- 录入 30 个节点：仍均匀
- 录入 100 个节点：球面密铺但节点中心不重叠
- 删除节点后位置自动重新分布

function nodeBrightnessLevel(n) {
  // n.count = 节点被引用篇数；n.role = primary/secondary/background
  let lv = 1;
