## 固定执行规则

1. 每次开工前先确认本次改动属于哪一类：数据层、同步层、UI 层、AI prompt、导出/导入、测试/文档。
2. 每次改动前列出影响面：会改哪些文件、哪些函数、哪些数据结构。
3. 每次改动前列出不动清单：明确哪些旧功能不能碰。
4. 每次改动后必须生成 `完成报告_*.md`，保存在项目根目录。
5. 完成报告必须包含：做了什么、验收标准、与预期相比、涉及文件、不动清单、测试说明。
6. 如果没有自动化测试，完成报告必须写明手动验收步骤。
7. 涉及数据写入、删除、导入、同步、迁移时，默认按 P0 风险处理。
8. 遵守真实问题日志里的执行准则：写代码前明确影响面、不动清单、测试用例、回滚路径；实现后生成完成报告。

## P0 数据安全问题

### P0-001 本地同步静默覆盖旧数据

- 问题：`parseMarkdownRecord()` 解析失败的字段曾经返回 `''` 或 `[]`，`syncFromLocalFolder()` 合并时会覆盖 IndexedDB 里的旧数据。
- 风险：用户只想改编辑评语，却因为节点格式异常导致完整 `nodes_hit` 被清空。
- 当前状态：已修复（2026-05-04 复查补强：列表段落出现部分解析异常时，整段不写回，避免半截覆盖旧数据）。
- 固定审核点：
  - 解析失败和用户主动清空必须区分。
  - 缺失字段不能覆盖旧字段。
  - 格式异常的列表段落不能返回空数组覆盖旧数组。
  - 同步计数不能把“没有可信字段”的文件算作已同步。

### P0-002 节点表与文章 analysis 不一致

- 问题：本地 Markdown 修改 `nodes_hit` 后，只更新文章 analysis，没有维护 `nodes` 表。
- 风险：资产详情显示新节点，但地图/复盘节点频次仍是旧数据。
- 当前状态：已修复。
- 固定审核点：
  - 任何修改 `analysis.nodes_hit` 的逻辑，都必须同步维护 `nodes` 表。
  - 删除文章、删除项目、导入、反向同步都要考虑节点引用清理。

### P0-003 同 URL 全局覆盖

- 问题：`dbSaveArticle()` 当前按 URL 全局覆盖旧记录。
- 风险：同一篇文章如果属于多个项目，后一次保存可能覆盖前一个项目里的记录。
- 当前状态：已修复（2026-05-04：`dbSaveArticle()` 改为同项目内 URL 唯一，不同项目允许保存同一 URL）。
- 审核判断：
  - 当前产品选择“同项目内 URL 唯一”。
  - 反向同步已改为 shortId 优先，URL 只在全库唯一时兜底，避免跨项目误写。
  - 2026-05-04 自查补强：备份导入会把备份 articleId 映射到实际保存后的 articleId，并重建该文章节点引用，避免 nodes 表引用旧 id。

### P0-004 删除后节点孤儿

- 问题：早期 `dbDeleteArticle(id)` 只删 `articles`，不清理 `nodes.articles` 引用。
- 风险：知识地图节点大小不准，复盘高频节点虚高。
- 当前状态：已修复。
- 固定审核点：
  - 删除单篇文章、删除项目、清空全部数据后，不能留下孤立节点引用。
  - 后续如果新增文章类型或灵感类型，删除逻辑也必须覆盖。

## P1 稳定性与一致性问题

### P1-001 扫描按钮文字闪变

- 问题：初始「重新扫描」，同步后变成「重新扫描本地文件夹」，挤压状态文字。
- 当前状态：已修复。
- 固定审核点：按钮在 loading 前后不应改变布局宽度。

### P1-002 分析流程可能被卡死

- 问题：`analyze()` 中曾发现先设置 `isAnalyzing = true`，无 API Key 时直接 return，可能不恢复状态。
- 当前状态：已修复。
- 审核判断：后续触碰 `analyze()` 时，任何早退路径都必须恢复 `isAnalyzing` 和按钮状态。

### P1-003 Raw JSON 固定 id 冲突

- 问题：分析页和资产详情页都可能存在 `rawToggleBtn` / `rawJson`，`getElementById()` 可能切错区域。
- 当前状态：已修复。
- 审核判断：后续触碰结果渲染时，继续使用局部 DOM 查找，避免重复 id。

### P1-004 HTML 属性转义不足

- 问题：`esc()` 曾只转义 `& < >`，进入 `data-*`、`title` 属性时可能被引号打断。
- 当前状态：已在节点反链改动中补强。
- 固定审核点：凡是写入 HTML 属性的动态内容，必须转义 `"` 和 `'`，或改用 DOM API。

### P1-005 手动模式只能用一次

- 问题：早期手动粘贴分析成功后隐藏 textarea，用户没有路径继续分析下一篇。
- 当前状态：已修复。
- 固定审核点：成功、失败、无 API Key 等路径都不能让用户卡死；必须能回到“分析新内容”。

### P1-006 跨标签页数据不同步

- 问题：早期设置页清空/导入数据后，已打开的 panel 不刷新。
- 当前状态：已修复为 `BroadcastChannel('zhijing_updates')`。
- 固定审核点：导入、清空、删除项目等跨页面操作要刷新 badge 和当前 tab。

### P1-007 捕获路径重复

- 问题：早期 `content.js` 自动捕获与 `background.js` 点击注入同时写 `lastArticle`，可能覆盖两次。
- 当前状态：已收敛，`content.js` 保留 `capture_request` 消息接口，主捕获由 `background.js` 点击注入触发。
- 固定审核点：后续改 DOM 选择器时，注意两处捕获函数仍有相似逻辑，避免行为分裂。

### P1-008 AI 非 JSON 错误提示

- 问题：小模型返回 markdown/说明文字/空响应时，直接暴露 JSON.parse 报错会很丑。
- 当前状态：已修复为 `parseAndValidate()` 统一提示。
- 固定审核点：新增 provider 或 prompt 时，保持错误提示可操作。

## P2 体验与边角问题

### P2-001 标签视觉不统一

- 问题：「新引入概念」使用 `.tag`，「标签」使用 `.asset-tag`，视觉语义不统一。
- 当前状态：已修复（2026-05-04 C4 首轮视觉回归：`.tag` 与 `.asset-tag` 收敛为同一套胶囊样式，资产列表保留可点击态）。
- 审核判断：留到 UI 打磨阶段，不应混在数据层改动里顺手改。

### P2-002 shortId 解析过松

- 问题：`a_b.md` 会取 `b` 当 shortId，理论上可能误匹配 id 末尾。
- 当前状态：已修复（2026-05-04：要求 shortId 至少 6 位字母数字；`a_b.md` 会被忽略并给出 warning；更长 id 取末 6 位匹配库内记录；shortId 碰撞时不兜底匹配）。
- 审核判断：反向同步不得使用少于 6 位或含非字母数字的 shortId 做匹配。

### P2-003 Obsidian Vault 导出未按项目分文件夹

- 问题：当前 A5 通过 front-matter 的 `project` 字段区分项目，所有文章导出到同一文件夹。
- 当前状态：可接受。
- 审核判断：如果用户要求更像 Vault，可新增按项目子文件夹导出。

### P2-004 控制台日志偏多

- 问题：`panel.js`、`background.js`、`content.js` 仍有较多 `console.log`。
- 当前状态：已修复（2026-05-07 T1：`panel.js` / `background.js` / `content.js` / `db.js` 加 DEBUG=false 和 `debugLog()`；诊断性 `console.log` 已收敛为 `debugLog`，`warn/error` 保留）。
- 审核判断：T1 之后不要新增裸 `console.log`，调试输出只走 `debugLog`；发布默认 `DEBUG=false`。

### P2-005 API Key 存储在 chrome.storage.sync

- 问题：API Key 跟随 Chrome 账号同步，介意安全的用户会担心。
- 当前状态：已修复（2026-05-04：`apiKeys` 改存 `chrome.storage.local`；旧 `sync.apiKeys` 会自动迁移到 local 后移除；provider/model 偏好仍保留 sync）。
- 审核判断：后续新增 provider 或设置项时，密钥继续只写入 `chrome.storage.local`，不要重新写回 `sync`。

### P2-006 正文只存前 2000 字

- 问题：`saveToLibrary()` 只保存 `article.body = articleText.slice(0, 2000)`。
- 当前状态：可接受但需注意。
- 审核判断：如果后续做 Markdown 编辑器或全文搜索正文，需要重新评估存储策略。

### P2-007 重新分析单篇

- 问题：资产详情页没有“用当前模型重新分析”。
- 当前状态：未修。
- 审核判断：属于 C 阶段编辑/二次创作能力，可后置。

### P2-008 资产 Tab 空计数徽标残留

- 问题：Day 1 视觉底座后，资产 Tab 的 `count-badge` 在计数为 0 时文本为空，但元素仍保留 padding 和金色背景，视觉上像“资产”后方多出一小条。
- 当前状态：不阻塞 Day 2，纳入 Day 5 下午功能清理日处理。
- 预定修法：`count === 0` 时将整个 `span.count-badge` 设为 `display: none`，有计数时再显示。

### P2-009 CSS 主题覆盖存在冗余

- 问题：Day 1 为避免大面积改坏旧 UI，采用末尾追加同 selector 覆盖旧 selector 的方式，功能正确但 CSS 存在冗余。
- 当前状态：不影响演示，不在 2.0 冲刺中重构。
- 预定修法：v2.1 统一整理 CSS，删除旧主题 selector 与重复覆盖。

## 每次完成报告必须回答的问题

1. 本次改动是否触碰数据写入、删除、导入、同步或迁移？
2. 是否可能造成旧数据被覆盖、丢失、重复、孤立引用？
3. 是否维护了 `articles` 与 `nodes` 的一致性？
4. 是否改变了旧 Markdown 格式或 JSON 备份格式？
5. 是否改变了 IndexedDB schema？如果改变，迁移路径是什么？
6. 是否影响 A1-A5 已有能力？
7. 自动化测试能覆盖什么？不能覆盖的手动验收是什么？
8. 如果三天后 Claude 只看完成报告，是否能复现验收？
