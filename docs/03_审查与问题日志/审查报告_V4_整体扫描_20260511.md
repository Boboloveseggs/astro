# 审查报告：V4 整体扫描

## 扫描范围

- 核心文件：`manifest.json`、`background.js`、`panel.js`、`panel.html`、`db.js`、`options.js`、`options.html`、`content.js`、`analyzer.js`
- 测试文件：`tests/test.html` 及现有测试套件
- 静态检查点：service worker、权限、数据备份、周复盘、资产批量操作、新手引导、知乎读取链路、地图/宇宙入口

## 测试结果

- Chrome Headless + DevTools Protocol：
  - `126 passed / 0 failed`

## 总体判断

当前没有发现会立刻破坏文章库的 P0 问题。主流程、测试、service worker 简化方向都还在轨。

但 V4 已经进入「功能密集堆叠期」，现在最大的风险不是某一个按钮，而是几类长期隐患：

- 部分新数据存在 `chrome.storage.local`，没有进入备份 / 恢复 / 清空链路。
- 分析完成后的入库失败会被静默吞掉，用户可能误以为保存成功。
- 面向用户的文案里仍残留“抓取”，和当前产品口径“读取当前页面内容 / 分析当前文章”不一致。
- 项目根目录残留大量 `.tmp_chrome_*` 临时目录，容易污染打包。
- `panel.js` 已经承担过多职责，后续每改一次都会更容易碰坏宇宙、复盘、资产、新手引导中的某一块。

## P1：建议优先修

### 1. 分析结果入库失败会静默吞掉

位置：

- `panel.js:1089-1121`

现状：

- `saveToLibrary()` 内部 `catch` 只 `console.warn('[知识图鉴] 入库失败:', e.message)`。
- `analyze()` 调用 `saveToLibrary()` 后没有检查返回值是否为 `null`。
- 如果 IndexedDB 写入失败、素材库写入失败，用户仍然已经看到了分析结果，但可能不知道这篇文章没有进入资产库。

风险：

- 用户以为已经保存，关闭面板后找不到记录。
- 这不是丢旧数据，但会造成新分析结果“看似完成、实际未入库”。

建议：

- `saveToLibrary()` 失败时向 UI 显示明确错误。
- `analyze()` 检查 `savedRecord`，为空时提示“分析已生成，但保存到资产库失败，请先导出或重试”。

### 2. 周复盘报告 / 写作指纹没有进入备份、恢复、清空链路

位置：

- `db.js:366-375`
- `db.js:258-267`
- `panel.js:5847-5857`
- `panel.js:6095-6101`
- `options.js:337-350`
- `options.js:398-403`

现状：

- `dbExportAll()` 只导出 `articles / nodes / clips / config`。
- 周报 `weeklyReports` 存在 `chrome.storage.local`。
- 写作指纹 `styleFingerprint` 也存在 `chrome.storage.local`。
- 设置页“清空全部数据”只调用 `dbClearAll()`，不会清掉周报和写作指纹。

风险：

- 用户导出备份再恢复后，历史周报和写作指纹消失。
- 用户清空文章库后，旧周报 / 旧指纹仍可能残留，造成“空库还有旧复盘”的错觉。

建议：

- 把 `weeklyReports`、`styleFingerprint` 纳入备份 JSON 的 `config` 或单独字段。
- `dbImportAll()` 恢复这些字段。
- “清空全部数据”同步清除 `weeklyReports` 和 `styleFingerprint`，但保留 API Key、主题、周目标等设置。

### 3. 根目录残留大量临时 Chrome Profile

位置：

- 项目根目录 `.tmp_chrome_*`

现状：

- 当前扫描到 64 个 `.tmp_chrome_*` 临时目录。

风险：

- 打 zip 或提交比赛包时容易把临时浏览器 Profile、缓存、IndexedDB 测试残留一起带进去。
- 备份体积膨胀，也可能混入调试痕迹。

建议：

- 增加固定清理脚本或打包脚本，排除 `.tmp_*`、`screenshots_*`、历史完成报告。
- 项目根目录保留源码、必要文档、必要截图即可。

## P2：中等优先级

### 4. Manifest 残留 `alarms` 权限

位置：

- `manifest.json:7-13`
- `background.js` 当前没有 `chrome.alarms`

现状：

- 周复盘自动生成已经迁移到 `panel.js`，service worker 不再用 alarm。
- `manifest.json` 仍声明 `"alarms"`。

风险：

- Chrome 审核 / Claude 审查会问“为什么需要这个权限”。
- 与“service worker 只负责开 panel + 读取文章”的目标不一致。

建议：

- 如果确认没有后续后台定时需求，移除 `"alarms"` 权限。

### 5. 用户可见文案仍残留“抓取”

位置：

- `panel.html:1998`
- `panel.js:747`
- `panel.js:752`
- `panel.js:775`
- `panel.js:779`
- `panel.js:816`
- `background.js:143`
- `background.js:151`

现状：

- 新手引导已经改成“读取当前页面内容 / 分析当前文章”。
- 但按钮 title、错误提示和后台错误仍有“抓取 / 抓到 / 可抓取”。

风险：

- 产品口径不统一。
- 面向普通创作者时，“抓取”比“读取当前页面内容”更生硬。

建议：

- 用户可见文案统一替换为“读取当前页面内容”“没有读到足够正文”“请确认正文已展开”。
- 代码内部注释可以保留技术词，但 UI 和错误提示不要出现。

### 6. 设置页已保存 API Key 的展示方式不够稳

位置：

- `options.js:271-275`

现状：

- 已保存密钥列表用 `innerHTML` 拼接。
- 显示 `key.slice(0, 8)`。

风险：

- 用户截图设置页时会露出 Key 前缀。
- 虽然普通 API Key 不会包含 HTML，但这里仍是用户输入进入 `innerHTML`，不够干净。

建议：

- 改成 DOM `textContent` 渲染。
- 展示文案改成“已保存”，最多显示末尾 4 位，不显示前 8 位。

### 7. “清空全部数据”的范围文案需要更精确

位置：

- `options.js:398-403`
- `db.js:258-267`

现状：

- UI 文案写“清空全部数据”，实际只清 `articles / nodes / clips`。
- 项目配置、周目标、主题、API Key、周报、写作指纹等不会全部清空。

风险：

- 用户对“全部数据”的理解和实际行为不一致。

建议：

- 要么改文案为“清空文章、素材和知识图谱”。
- 要么提供两个按钮：“清空作品数据”和“恢复出厂设置”。

## P3：后续维护项

### 8. `panel.js` 已经过大

现状：

- `panel.js` 约 287KB，承担分析、资产、地图、宇宙、新手引导、复盘、周报、追问、批量操作等职责。

风险：

- 后续任何小修改都可能误伤已经定型的宇宙 / 知识球视觉。
- Claude 审查会越来越难定位“这次到底改了什么”。

建议：

- 短期不要大拆。
- 后续只在稳定期拆模块：
  - `panel_onboarding.js`
  - `panel_assets.js`
  - `panel_review.js`
  - `panel_universe.js`
- 拆之前必须先保证测试和截图基线更完整。

### 9. 版本号仍是 `2.0.0`

位置：

- `manifest.json:4`

现状：

- 项目实际已经是 V4 / V21-V23 自用版迭代状态。
- Manifest 仍是 `2.0.0`。

风险：

- 打包参赛或给用户安装时，版本识别混乱。

建议：

- 发布前统一版本号，比如 `4.0.0` 或 `2.3.0`，并写清 changelog。

## 通过项

- `background.js` 没有 `importScripts`，service worker 没有重新引入复杂依赖。
- `chrome.alarms` 没有实际调用。
- 新手引导默认不强制弹出，问号入口逻辑符合最近要求。
- 地图入口逻辑保留“进入地图默认宇宙全貌”。
- 当前测试全过：`126 passed / 0 failed`。
- 没有在源码中发现实际乱码，之前 PowerShell 显示乱码只是控制台编码问题，文件本身是 UTF-8。

## 建议下一步

建议先修 P1 的两项：

1. 入库失败给用户明确提示，避免“分析完成但没保存”的隐性损失。
2. 把 `weeklyReports / styleFingerprint` 纳入备份、恢复、清空规则，避免周报历史和写作指纹游离在主数据体系之外。

这两项都不需要动宇宙视觉、不需要改 Markdown 格式，也不需要改 IndexedDB schema。
